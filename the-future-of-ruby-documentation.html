And most of these expectations fall on one project: RDoc. RDoc has been the backbone of Ruby documentation for more than 20 years. It generates the official documentation at docs.ruby-lang.org. It also generates the data the ri command-line tool reads, so when you look up a method in the terminal, that's RDoc behind the scenes. So if we want to make Ruby's documentation work better for AI, RDoc is where the work has to happen.

But RDoc isn't ready for that yet.

But honestly, RDoc isn't ready for that future yet

What AI needs from documentation

  • Clear, accurate documentation
  • Simpler content formats — less context overhead (currently Markdown)
  • Clear intent — type signatures tell it what code expects
07
AI needs a few things from documentation. First, accuracy — AI trains on docs and builds assumptions based on them. It also grabs docs when making plans, and references them when generating code. Second, the doc should be presented in simpler content formats — ones that don't waste context on markup noise. Right now that's Markdown, but the specific format could change. Third, clear intent — type signatures give AI structured information about what a method expects and returns. And how different methods could interact with each other.

What AI needs from tooling

  • Quick, deterministic feedback — coverage checks, reference validation
  • Live preview — see changes without regenerating everything
08
That's the content side. But AI also needs good tooling around documentation so it can help developers contribute to them. When an agent writes or updates docs, it needs fast feedback — so it knows if anything is missing documentation, or if the references are incorrect. It can also use incrementally generated docs to verify changes quickly, instead of generating the whole project's documentation all over again.

All of these help human developers too.

These aren't AI requirements — they're good documentation requirements.

09
Here's the thing. Most of what I just described — accurate docs, type signatures, fast feedback — these aren't just AI requirements. They're what makes documentation better for everyone. Improving these things now just creates more leverage because AI amplifies the benefit.

RDoc wasn't delivering

  • Parser couldn't handle some newer Ruby syntax
  • Incomplete Markdown support
  • No RBS type signatures
  • No live preview
  • Dated theme, no mobile support

Parser

The new parser

  • RDoc's Ruby parser used Ripper — couldn't handle modern syntax and hard to maintain
  • Tomoya Ishida (@tompng) rewrote it on Prism, fixing many issues along the way
  • Fixed 7 long-standing bugs — some open since 2016
15
That's the new look. Now let's talk about what's under the hood. RDoc's Ruby parser was built on Ripper. It couldn't handle modern Ruby syntax and was hard to maintain. Tomoya Ishida — Ruby committer and RDoc maintainer, created a new parser based on Prism and fixed many issues along the way, some were reported even 10 years ago.

Parser

Bug fix example: __FILE__

Source: # Returns __FILE__ as a string

Before

Returns  FILE  as a string

After

Returns __FILE__ as a string

Same fix for __send__, __dir__, and similar identifiers.

16
The old formatter saw underscores and assumed they meant emphasis. So __FILE__ in a doc comment became italic-bold FILE. The new parser understands these are Ruby keywords, not formatting, so can render it correct. The same fix handles __send__, __dir__, and any double-underscore identifier.

Parser

What the rewrite changed

  • Roughly half the code (1,243 vs 2,241 lines)
  • ~30% faster (0.69s vs 0.97s on 112 files)
  • New Ruby syntax is handled by Prism — not by RDoc
16
After the rewrite, the new parser is roughly half the code — 1,243 lines versus 2,241. We benchmarked both on RDoc's own codebase — 112 Ruby files — and the new parser came out about 30% faster. But most importantly, when Ruby adds new syntax, Prism already knows how to parse it. We don't have to manually teach RDoc about every new feature anymore.

Parser

Bug fix example: endless methods

class Foo # open Foo def bar = 42 # open bar — where's end? end # this closes bar, not Foo class Baz; end # Ripper thinks we're still inside Foo

Ripper

class Foo
— method bar
— class Baz ← wrong

Prism

class Foo
— method bar
class Baz
17
Here's one example: The old Ripper parser works on tokens — it tracks where methods end by looking for the end keyword. But after endless methods were introduced, they broke that assumption. So in this example, the Ripper-based parser would incorrectly consume the end keyword of the Foo class, thinking it's the end for the method, and cause the Baz class to be considered nested inside Foo. With Prism, the parser already knows how to handle method and class boundaries. So RDoc can easily know there's just one method inside the Foo class. And Baz is outside of its scope.

Markdown

Writing Markdown in RDoc

  • Markdown is widespread — GitHub, Stack Overflow, every doc tool uses it
  • It should be the default in RDoc
  • RDoc has supported it since 2011, but the parser was buggy and incomplete
18
Next, let's talk about Markdown. Markdown is everywhere — GitHub, Stack Overflow, basically every documentation or developer tool uses it. It should be the default in RDoc. RDoc has actually supported Markdown since 2011. But the support was incomplete and buggy — features like strikethrough, tables, and heading anchors were all half working. If you're used to writing Markdown on GitHub, you'd be frustrated by RDoc's Markdown support pretty quickly.

Markdown

RDoc markup vs Markdown

RDoc markup

= Heading *bold* and _italic_ - item 1 - item 2 indented code block {link text}[http://example.com]

Markdown

# Heading **bold** and *italic* - item 1 - item 2 ```ruby fenced code block ``` [link text](http://example.com)

But most people don't know how to write it.

19
So what do we write with RDoc now? RDoc actually has its own markup language, and it's still the default instead of Markdown. I believe many of you didn't even know it exists. Look at the syntax — equals sign for headings, curly braces for links, indentation for code blocks. If you need to switch between the 2 languages, it's easy to mistakenly write markdown inside RDoc and it's very annoying. And because RDoc has to support both formats, fixing Markdown bugs is harder than it should be.

Markdown

Example: [text](url)

Markdown

↓

RDoc
(no rewrite)

↓

TIDYLINK node
↓

<a href="rdoc.sh">RDoc </a>
20
So why are Markdown bugs harder to fix than they should be? Let's trace one link through the pipeline. When you write a Markdown link, the Markdown parser rewrites it into RDoc's link syntax before it even lands in the document tree. That's the second box. And that's also where RDoc markup enters the pipeline directly — no rewrite needed. From there both paths run the same. The shared inline layer tokenizes it into a TIDYLINK node, and ToHtml turns the node into an anchor tag. So Markdown is really an extra transpiler bolted onto the front of RDoc's pipeline. Every Markdown construct pays this hop, and any fix has to keep RDoc markup working too. That's why small Markdown bugs take disproportionate effort.

Server mode

rdoc --server

  • Changes reflected in browser automatically
  • Zero external dependencies — uses Ruby's TCPServer
  • Incremental re-parsing — shorter response, not instant (yet)
  • Run rdoc --server in any gem using RDoc to preview its docs
  • make html-server in ruby/ruby — the same flow for Ruby core docs
26
The server mode has zero external dependencies — it uses Ruby's built-in TCPServer. It has a background watcher monitoring files and only re-parses the ones that need to change. The latency is between 100 to 200 milliseconds right now. With future improvements on parsing and rendering performance, I hope we can make it feel instant someday. Any gem that uses RDoc gets this out of the box — just run rdoc --server in your project and you can preview your documentation changes the same way. And I integrated this into Ruby itself — so now you can preview core documentation the same way. This lowers the barrier to contribute to Ruby's documentation.

There are three areas I want to work on next. First is contributing — making it easier to write and maintain Ruby documentation, both for Ruby core and for gem docs. Second is AI skills — RDoc related skills that help agents use the tool correctly. Third is LLM-ready docs — reducing the cost for agents to read Ruby documentation. I'll walk through each one.

Contributing

Setting good standards

  • RBS signatures — structured, machine-readable contracts
  • Revisiting supported directives for clarity
  • Whether users adopt a type checker is their choice — RDoc supports both
30
RBS signatures are a start — structured, machine-readable contracts in the source. We'll revisit supported directives to make them clearer. And I want to be explicit: whether users adopt a type checker is their choice. RDoc will support both typed and untyped codebases.

Contributing

Fully migrate to Markdown

  • Automatic RDoc markup → Markdown translation
  • Migrate Ruby core's documentation to Markdown
  • Change RDoc's default markup to Markdown

A large effort — but AI-assisted tooling can help us get there faster.

30
One of the long-term goals to simplify doc contribution in Ruby is to fully migrate RDoc to use Markdown. So everyone can just use the language they know best to contribute to documentation. But to achieve that, it means building an automatic translator from RDoc markup to Markdown, migrating Ruby core's documentation — which is massive — and eventually making Markdown the default. This is a lot of work. But with AI-assisted tooling, I'm hoping we can get there faster than it would have taken a few years ago.

Contributing

Make documentation reflect code design

  • New RDoc annotations (not RBS): @abstract, @override, @interface
  • Link abstract methods to their implementations
  • Show override relationships across class hierarchies

Design intent in the docs — useful for both humans and AI.

31
Another thing I want to explore is documentation annotations that reflect code design. Concepts like abstract methods, overrides, and interfaces are commonly seen in typing languages. But they're useful for documentation too regardless of whether you use a type checker. The relationships these annotations describe will be helpful for both humans and AI

Contributing

Example: @abstract + @override

class Formatter # @abstract def convert(doc) raise NotImplementedError end end
class ToHtml < Formatter # @override def convert(doc) = ... end
class ToMarkdown < Formatter # @override def convert(doc) = ... end
Formatter#convert abstract
⟷
ToHtml#convert override
ToMarkdown#convert override
32
Here's what it could look like. Let's say you have a parent class defining an abstract method that child classes should override. You can use abstract and override annotations to provide that information. Then in the generated docs, RDoc can list all of the implementations at the Formatter class' page. And each override would also have UI links back to the parent method. Combining annotations with type signatures will allow us create more expressive documentation that benefits all consumers.

AI skills

How AI can help projects adopt or use RDoc

  • Set up RDoc in a project — config, exclusions, conventions
  • Keep up with new features — RBS, annotations, server mode
  • Write the docs themselves — and let the user review
33
Now let's talk about how AI can help projects adopting/using RDoc There are a few things an AI agent could do for you. First, setting RDoc up in a project. RDoc has many configuration options, such as the main page, the markup language, files to exclude...etc.. An agent could help you figure these out interactively. Second, keeping up with new features. As we improve RDoc and keep adding new capabilities — RBS support and annotations. If you don't follow RDoc development closely, you probably don't know what's available. An agent could bridge that gap. Third, they can write the docs themselves. A lot of developers already let AI write code for them. We should be able to do the same for documentation. You just need to review the results.

34
To make that kind of experience work, we'd need a way to give agents the right instructions. That's what an AI skill is for, in case you haven't used one before. A skill is just a Markdown file following the SKILL.md standard. It has a name, a description of when to use it, and instructions for the agent. When the agent encounters a relevant task — like writing Ruby documentation — it reads the relevant skills and follows the instructions.

AI skills

What the RDoc skill could teach

  • Preview workflow — rdoc --server and incremental updates
  • Writing RBS signatures — inline #: or a sig/ directory
  • Using annotations — @abstract, @override, @interface
  • Checking coverage — rdoc -C to find undocumented classes, methods, constants, and attributes

Everything coming in RDoc 8.0, discoverable for agents.

35
So what would an RDoc skill actually teach the agent? It would cover the core workflows. Things like using the server for live preview, writing type signatures either inline or in a sig directory, and using the new design annotations we just talked about. It would also cover how to check documentation coverage — running rdoc dash C to find undocumented classes, methods, constants, and attributes before a PR gets merged. With RDoc skills, everything coming in RDoc 8.0 will become discoverable for agents, without users having to explain them manually.

But how do users install it?

35
So that's what the skill could teach. Writing the skill is actually the easier part. The harder question is how users get it onto their system. An agent doesn't pull skills out of thin air. Someone has to install them, keep them up to date, and maybe pin them to specific versions. Honestly, there's no clean answer yet. Every agent has its own way of doing this, and none of them are standardized. So before we ship an RDoc skill, we need to figure out which approach fits best. That's something we'll work on after 8.0 ships.

Contributing

Better tools for writing docs

  • For humans: rdoc --server (live preview)
  • For AI agents: rdoc -C (coverage — what's missing)
  • For AI agents: ri --format=markdown (query docs directly)

For humans, server mode gives you live preview while writing. For AI agents, rdoc -C gives coverage reports — what's undocumented, what has broken references. And ri --format=markdown lets agents query Ruby documentation directly in a format they're fluent in.

36
Now let's talk about making the documentation ready for AI consumption. docs.ruby-lang.org is how most people read Ruby's documentation — and it's also where agents go when they need to look something up. Today, it's all HTML. So I ran Array.html through a tokenizer — the same kind that AI models use. The raw HTML is 183,000 tokens. That's almost the entire 200k context window, for a single class. Strip it down to Markdown and it drops to 69,000. Plain text is 58,000. So roughly two thirds of those HTML tokens are pure overhead — navigation chrome, CSS classes, anchor scaffolding. For an agent, that's a lot of wasted context.

LLM-ready docs

OK, ship Markdown. How do agents find it?

  • Generate Array.md next to Array.html?
  • Let agents scrape HTML and convert on the fly?
  • Point them somewhere explicit — a directory they can crawl?

llms.txt was proposed as the answer.

37
So shipping Markdown seems like the fix. But that's only half of it — the other half is discovery. We could generate a .md file next to every .html. But if agents don't know to look for it, they just keep fetching HTML. We could let them scrape and convert every page themselves. But then every agent pays the same conversion cost, and the result depends on whatever converter they happen to use. Or we can point them somewhere explicit. That's what llms.txt was proposed for.

LLM-ready docs

llms.txt: a sitemap for AI

  • A single /llms.txt at the site root
  • Lists canonical Markdown paths for every doc page
  • Agents read it first, then fetch only the .md files they need
# Ruby docs

- [Array](Array.md): core class
- [Hash](Hash.md): core class
- [String](String.md): core class
38
Let me briefly explain how llms.txt works, in case you haven't seen it. It's a single plain text file at the root of your docs site, similar to robots.txt. Inside, you list the most important Markdown URLs with a short description of each. The idea is that agents fetch llms.txt first to get the lay of the land, then pull just the .md files they need — skipping the HTML chrome entirely. It's a simple idea, and cheap to implement. The real question is whether anyone actually respects it.

39
So I went and checked who actually ships llms.txt today. Let's start with Rust — they don't ship one. An RFC to add LLM-friendly output to rustdoc was closed without merging. The maintainers thought LLM tooling is still moving too fast to commit to a stable format. Python, Node, MDN, and Ruby don't ship one either. None of their robots.txt files even mention llms.txt or AI crawlers. The one exception is the Elixir ecosystem. ExDoc shipped llms.txt generation as a default in version 0.40 earlier this year, which means every project that rebuilds its docs with a recent ExDoc version gets one automatically. Phoenix LiveView has it. Elixir's main branch docs have it. The next Elixir release will have it too. What's interesting is how they did it. They didn't ask each project to opt in. They made it a default in the doc generator itself, so adoption propagates with normal release cadence. But they are still the only ecosystem doing this. So we have a proposed standard, one ecosystem that committed hard, and everyone else waiting.

LLM-ready docs

Going back to RDoc

  • Most communities haven't adopted llms.txt yet
  • Elixir is the exception — ExDoc ships it by default, and hexdocs.pm propagates on every rebuild
  • We'll monitor Elixir's rollout, and prototype a .md + llms.txt solution for RDoc in the future
40
So what does all of this mean for RDoc? The honest answer is that we'll watch and learn before committing. Most communities still haven't adopted llms.txt, and the ones that considered it — like Rust — have pushed back. So rushing to ship our own version doesn't make sense yet. The exception is the Elixir ecosystem. ExDoc made llms.txt a default in the doc generator, so it propagates to every hexdocs.pm project on the next rebuild. That's a clever distribution model, and it gives us a real case study to observe. So our plan is to monitor how Elixir's rollout plays out. Whether agents actually start fetching llms.txt from hexdocs.pm in practice. Whether other ecosystems follow. Whether the format evolves. And in parallel, we'll prototype a Markdown output plus llms.txt solution for RDoc as future work. Not something we commit to shipping with 8.0, but something we have ready when the pattern settles.

Summary

  • What makes docs useful to AI is what makes them useful to developers
  • RDoc 8.0 ships the fixes — new parser, Markdown improvements, RBS, server mode
  • Still ahead — Markdown migration, design annotations, an RDoc skill, and a .md + llms.txt prototype

That means accurate docs, simpler content formats, type signatures and annotations for clear intent, and good tooling for fast feedback. Second, RDoc 8.0 gets us a lot closer to that goal. It ships with the new Prism parser, Markdown improvements, RBS type signatures, and server mode. Third, there's still a lot ahead.
